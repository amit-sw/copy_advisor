Direct Quote APIs and Setup

The notebook begins like a field technician opening a locked service panel: get credentials, confirm the wiring, then prove the data path works before asking an LLM to narrate anything.

The first architecture choice is to separate access from analysis

This notebook does not start with a finance question. It starts by installing dependencies, loading secrets, and binding a named environment variable for each API key.

Why that matters: if live market access is not stable on its own, every later LLM answer is built on sand. Software engineers care because this is dependency hygiene; product partners care because it explains why “the model failed” is often really “the integration was never ready.”

Think of Alpha Vantage as calling the exchange desk directly, while package-based Yahoo Finance is like using a well-trained broker. Both can answer “what is the stock doing,” but one exposes the raw request more clearly and the other packages richer metadata.

Code 
def get_stock_quote(symbol: str) -> dict:
    """Fetch the latest stock quote for ``symbol`` using the Alpha Vantage API."""
    params = {"function": "GLOBAL_QUOTE", "symbol": symbol, "apikey": ALPHAVANTAGE_API_KEY}
    response = requests.get(ALPHAVANTAGE_API_URL, params=params, timeout=10)
    response.raise_for_status()
    data = response.json().get("Global Quote", {})
    if not data:
        return {"symbol": symbol.upper(), "price": None}
    return {
        "symbol": data.get("01. symbol", symbol.upper()),
        "price": float(data.get("05. price", "0") or 0.0),
        "change_percent": data.get("10. change percent"),
    }
Plain English

Define one helper whose whole job is “give me the latest quote for this symbol.”

Build a compact request payload with the exact Alpha Vantage function name the provider expects.

Fail fast if the network call itself is broken.

Extract only the `Global Quote` slice instead of making every caller interpret provider-specific JSON.

If no quote comes back, return the same schema anyway so downstream code stays predictable.

Normalize provider field names into business-friendly keys such as `symbol`, `price`, and `change_percent`.

The notebook is teaching contract design, not just ticker lookup

A schema is the quiet hero here. `get_stock_quote()` returns a Python dictionary; `get_stock_quote_yf()` returns JSON text. That difference signals who the expected consumer is and how much cleanup still remains.

def get_stock_quote_yf(symbol: str) -> str:
    """Get current stock price and basic information."""
    try:
        stock = yf.Ticker(symbol)
        info = stock.info
        hist = stock.history(period="1d")
        if hist.empty:
            return json.dumps({"error": f"Could not retrieve data for {symbol}"})

        current_price = hist['Close'].iloc[-1]
        result = {
            "symbol": symbol,
            "current_price": round(current_price, 2),
            "company_name": info.get('longName', symbol),
            "market_cap": info.get('marketCap', 0),
            "pe_ratio": info.get('trailingPE', 'N/A'),
            "52_week_high": info.get('fiftyTwoWeekHigh', 0),
            "52_week_low": info.get('fiftyTwoWeekLow', 0)
        }
        return json.dumps(result, indent=2)
Plain English

Wrap the provider logic in a `try` block because package-backed integrations can fail in several places.

Ask Yahoo Finance for both identity information and a one-day price history.

Return an explicit error payload if the history table is empty instead of pretending a price exists.

Use the latest closing value as the current price for this notebook’s purposes.

Bundle price, company name, valuation clues, and range data into one result object.

Serialize that object to formatted JSON so it is easy to display or pass along as text.

Function Tools Without MCP

This module is the customs checkpoint of the notebook: the model can request a stock quote, but Python still checks the paperwork, executes the call, and decides when the conversation is done.

The tool schema explains what the model is allowed to ask for

If Module 1 separated live facts from model narration, Module 2 explains how the model gets permission to reach those facts. The tool declaration is like a typed service contract: name the function, describe its purpose, and declare the one argument that matters.

Code 
response = client.chat.completions.create(
    model=model,
    messages=messages,
    tools=[
        {
            "type": "function",
            "function": {
                "name": "get_stock_quote",
                "description": """Fetch the latest stock quote for ``symbol`` using the Alpha Vantage API.""",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "symbol": {
                            "type": "string",
                        },
                    },
                    "required": ["symbol"],
                },
            },
        },
    ],
    temperature=1,
)
Plain English

Ask the chat-completions API to answer in the context of the current message history.

Advertise one callable tool schema instead of hoping the model guesses how to access quote data.

Tell the model the function is named `get_stock_quote`, which is the Python helper it can indirectly request.

Explain what the tool is for in natural language so the model knows when it is relevant.

Constrain the arguments to an object containing a required string field called `symbol`.

Leave the model free to decide whether it needs the tool, but not free to invent a different function contract.

`answer_question()` is the actual orchestrator

A orchestrator is like an air-traffic controller: it does not fly the plane, but it decides what happens next and prevents collisions. Here, that role belongs to Python, not the model.

Code

Plain English

Keep looping until the model either finishes normally or the notebook decides the conversation has gone too far.

The code is defensive because model output is not a perfect API client

The model may emit malformed JSON, odd capitalization, or an older function-calling shape. The notebook compensates by parsing carefully, looking for `symbol` case-insensitively, and serializing tool results back into the conversation as explicit messages.

MCP Server Calls and Introspection

The last module switches metaphors from “private service desk” to “public transit network”: instead of exposing one local function, the notebook connects to a remote MCP server and then inspects the station map directly.

MCP changes the integration boundary

In Module 2, the notebook advertised a local function to the model. Here, it hands the model a remote MCP server endpoint, which means the tool surface lives outside the notebook and can be discovered through protocol calls.

Code 
response = client.responses.create(
    model="gpt-5-mini",
    input="Get the latest stock quote for AAPL using the Alpha Vantage MCP server. It is okay to get delayed or most recent close.",
    tools=[
        {
            "type": "mcp",
            "server_url": f"https://mcp.alphavantage.co/mcp?apikey={ALPHAVANTAGE_API_KEY}",
            "server_label": "alphavantage",
            "require_approval": "never"
        }
    ]
)
Plain English

Use the Responses API instead of the earlier chat-completions path for this remote-tool example.

Ask a concrete stock question that clearly needs fresh market data.

Expose a tool of type `mcp`, which tells the client this capability comes from a protocol server rather than a local Python helper.

Point the client at the Alpha Vantage MCP endpoint and include the provider key in the server URL.

Give the remote server a readable label so it has an identity in the tool configuration.

Allow the call without a separate approval step inside this notebook context.
