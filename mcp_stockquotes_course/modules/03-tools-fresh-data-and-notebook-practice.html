MCP Server Calls and Introspection

The last module switches metaphors from “private service desk” to “public transit network”: instead of exposing one local function, the notebook connects to a remote MCP server and then inspects the station map directly.

MCP changes the integration boundary

In Module 2, the notebook advertised a local function to the model. Here, it hands the model a remote MCP server endpoint, which means the tool surface lives outside the notebook and can be discovered through protocol calls.

Code 
response = client.responses.create(
    model="gpt-5-mini",
    input="Get the latest stock quote for AAPL using the Alpha Vantage MCP server. It is okay to get delayed or most recent close.",
    tools=[
        {
            "type": "mcp",
            "server_url": f"https://mcp.alphavantage.co/mcp?apikey={ALPHAVANTAGE_API_KEY}",
            "server_label": "alphavantage",
            "require_approval": "never"
        }
    ]
)
Plain English

Use the Responses API instead of the earlier chat-completions path for this remote-tool example.

Ask a concrete stock question that clearly needs fresh market data.

Expose a tool of type `mcp`, which tells the client this capability comes from a protocol server rather than a local Python helper.

Point the client at the Alpha Vantage MCP endpoint and include the provider key in the server URL.

Give the remote server a readable label so it has an identity in the tool configuration.

Allow the call without a separate approval step inside this notebook context.

The notebook refuses to treat MCP as a black box

This is the strongest “why” in the whole notebook. Instead of stopping after `response.output_text`, the author uses raw JSON-RPC to list tools and inspect one tool’s parameter contract directly.

Code 
payload = {
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/list"
}
response = requests.post(url, json=payload)
data = response.json()
if isinstance(data, str):
    data = json.loads(data)

tools = data["result"]["tools"]
Plain English

Construct a protocol message asking the server to enumerate its available tools.

Post that message directly to the MCP endpoint, bypassing any high-level abstraction for a moment.

Parse the JSON response, but stay defensive in case the payload arrives as a string.

Pull the `result.tools` list out as a discoverable catalog of what the server actually offers.

Code 
payload = {
    "jsonrpc": "2.0",
    "id": 2,
    "method": "tools/call",
    "params": {
        "name": "TOOL_GET",
        "arguments": {
            "tool_name": "GLOBAL_QUOTE"
        }
    }
}

resp = requests.post(url, json=payload)
outer = resp.json()
inner_text = outer["result"]["content"][0]["text"]
data = ast.literal_eval(inner_text)
Plain English

Build a second protocol message that calls a specific metadata tool on the server.

Ask for details about `GLOBAL_QUOTE`, which is the quote capability the course has already seen in raw API form.

Read the nested result structure out of the server response.

Convert the returned text payload into a Python object so the notebook can display the tool contract clearly.

Inspection is a product and engineering skill, not just a coding trick

The notebook renders the listed tools into Markdown because human-readable contracts reduce ambiguity. Engineers can debug faster, and product partners can reason about what the model is genuinely able to access.
