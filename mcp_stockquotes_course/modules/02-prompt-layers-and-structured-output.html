Function Tools Without MCP

This module is the customs checkpoint of the notebook: the model can request a stock quote, but Python still checks the paperwork, executes the call, and decides when the conversation is done.

The tool schema explains what the model is allowed to ask for

If Module 1 separated live facts from model narration, Module 2 explains how the model gets permission to reach those facts. The tool declaration is like a typed service contract: name the function, describe its purpose, and declare the one argument that matters.

Code 
response = client.chat.completions.create(
    model=model,
    messages=messages,
    tools=[
        {
            "type": "function",
            "function": {
                "name": "get_stock_quote",
                "description": """Fetch the latest stock quote for ``symbol`` using the Alpha Vantage API.""",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "symbol": {
                            "type": "string",
                        },
                    },
                    "required": ["symbol"],
                },
            },
        },
    ],
    temperature=1,
)
Plain English

Ask the chat-completions API to answer in the context of the current message history.

Advertise one callable tool schema instead of hoping the model guesses how to access quote data.

Tell the model the function is named `get_stock_quote`, which is the Python helper it can indirectly request.

Explain what the tool is for in natural language so the model knows when it is relevant.

Constrain the arguments to an object containing a required string field called `symbol`.

Leave the model free to decide whether it needs the tool, but not free to invent a different function contract.

i
Why this matters

Software engineers should read this as interface design. Product partners should read it as operational scope. In both cases, the point is the same: the model can only ask for what the app explicitly exposes.

`answer_question()` is the actual orchestrator

A orchestrator is like an air-traffic controller: it does not fly the plane, but it decides what happens next and prevents collisions. Here, that role belongs to Python, not the model.

Code 
while True:
    loop_count += 1
    if loop_count > loop_count_limit:
        raise Exception("Loop count exceeded")

    response = get_completion(AI_MODEL, messages)
    print(f"\n\n*** For input {messages}\n*** Received response {response}\n\n")

    choice = response.choices[0]
    msg = choice.message
    finish = choice.finish_reason

    if finish == "stop":
        return msg.content
Plain English

Keep looping until the model either finishes normally or the notebook decides the conversation has gone too far.

Increment a loop counter so runaway tool usage becomes visible and stoppable.

Ask the model for the next step using the current messages plus any prior tool results.

Log the request and response for debugging, which is especially useful when AI behavior feels “mysterious.”

Read the model’s finish reason because that is how the API signals whether it wants to stop or call a tool.

Return plain content only when the model declares it is done.

S

`stop`

The model has enough context and is returning a final answer.

T

`tool_calls`

The model is asking the notebook to execute one or more tools before it can continue.

L

Legacy `function_call`

The notebook also supports an older response shape, which is a practical compatibility choice.

The code is defensive because model output is not a perfect API client

The model may emit malformed JSON, odd capitalization, or an older function-calling shape. The notebook compensates by parsing carefully, looking for `symbol` case-insensitively, and serializing tool results back into the conversation as explicit messages.

Code 
try:
    args = json.loads(arg_json) if isinstance(arg_json, str) else (arg_json or {})
except Exception:
    args = {}

if isinstance(args, dict):
    symbol = args.get("symbol")
    if symbol is None:
        for k in list(args.keys()):
            if k.lower() == "symbol":
                symbol = args[k]
                break
Plain English

Try to parse the tool arguments as JSON, but do not assume the model emitted perfect syntax.

Fall back to an empty dictionary if parsing fails so the notebook stays in control.

Check whether the parsed value is really a dictionary before trying to read named fields from it.

Look for `symbol` directly first because that is the preferred contract.

If it is missing, search again case-insensitively so small formatting drift does not break the flow.

!
The deeper why

The notebook is teaching a realistic AI architecture principle: models are probabilistic planners, so application code must still own validation, retries, shape normalization, and termination rules.
