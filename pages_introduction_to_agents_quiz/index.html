02

Monolith Strengths and Limits

The deck does not dismiss the monolith outright. It explains why it works at first, then shows where it begins to fail.

Study

The monolith is attractive because it is simple, but simplicity degrades as scope grows

The monolithic prompt is structured like one giant instruction bundle: branding and tone at the top, policy sections for different use cases, and tool instructions for cases like shipping checks. That can work surprisingly well for straightforward requests. The trouble starts when context windows are constrained, instructions pile up, and the system must support more use cases or multi-step actions.

  • The monolith has real advantages: fast setup, decent speed, and acceptable behavior on simple requests.
  • The deck specifically calls out limited context windows, around 8,000 tokens in the example era.
  • More directions can increase mistakes and make prompts harder to reason about.
  • Maintenance pain grows when new use cases keep getting added or users require several linked actions.

03

Workflow Agents as Modular Engineering

The deck's answer to monolith sprawl is straightforward: divide the work, specialize the roles, and manage the flow between them.

Study

Workflow agents are presented as software modularity applied to AI behavior

The presentation reframes the problem using standard engineering instincts: divide and conquer, break problems into steps, and prefer multiple specialists over one overloaded generalist. In the workflow pattern, a front-desk or triage agent classifies incoming work, then hands control to the appropriate specialist. The surrounding workflow coordinates information and control flow.

  • Each agent is supposed to do one job really well.
  • Triage is explicit: one component decides where a request should go next.
  • This architecture makes changes more local, debugging easier, and growth less disruptive.
  • The modular pitch is not just conceptual; it is a direct answer to the scaling problems from the monolith section.

05

Shared State, Routing, and Tools

The workflow only works because the agents are coordinated through shared state, conditional routing, and explicit control-flow rules.

Study

These agents do not chat freely with each other; they coordinate through orchestration state

The communication slides emphasize that this design is indirect. A classifier writes fields like category into shared state, a router reads that state, and the chosen downstream agent reads the same messages before writing a response back. The control-flow code shows the same idea in graph form: start goes to classifier, the classifier uses conditional edges, downstream specialists end the workflow, and LangGraph provides the graph machinery. Tool usage fits into this same structure, even though the example status agent used a more brute-force version.

  • The deck says this is specific to LangGraph choices, but also a good design pattern.
  • One shared state object solves the parameter-passing problem across functions.
  • The example AgentState includes messages, response, and category.
  • The tool-usage slide distinguishes a brute-force approach from the more typical conditional tool-calling pattern supported by LLM APIs.
