Study

  • The motivating use case is customer support, where requests can vary by intent and tone.
  • The opening is doing architectural setup, not offering a vendor comparison.

02

Why the Monolith Stops Scaling

The deck shows why one giant prompt feels attractive at first, then details the practical limits that make it fragile.

Study

The monolith is appealing because it is simple, but simplicity collapses as requirements spread

The first attempt is a single support prompt that contains brand tone, multiple policy sections, and access to tools. That design is easy to start with and can work for straightforward requests, but the deck argues that its weaknesses emerge as the system grows. Limited context windows, instruction overload, maintenance strain, and multi-step tasks all push against the one-big-prompt strategy.

  • The monolith bundles header guidance, policies, and tooling in one place.
  • Its strengths are speed, simplicity, and decent performance on easy tasks.
  • Its limitations show up under scale, complexity, and ongoing maintenance.
  • The problem is not just prompt length; it is operational manageability.

03

Workflow Agents as Modular Design

The solution is framed as modular engineering: split responsibilities, route work deliberately, and keep each agent narrow enough to be understandable.

Study

The workflow architecture is a divide-and-conquer answer to the monolith

Once the monolith becomes hard to trust, the deck proposes modular engineering. Instead of one agent trying to embody every behavior, a workflow decomposes the job into distinct roles. A front-desk agent performs triage, specialists handle their own cases, and the workflow layer becomes responsible for moving information and control between them.

  • The deck describes the shift as divide-and-conquer and a multi-step problem-solving approach.
  • Specialists are favored over one overloaded generalist.
  • The workflow itself becomes a design artifact, not just an implementation detail.
  • Benefits include easier updates, clearer debugging, and simpler extension with new specialists.

05

Shared State and Tool Decisions

The closing material explains how the workflow actually coordinates work: a shared state object, explicit routing rules, and tool calls handled as part of orchestration.

Study

The agents cooperate through a common contract rather than improvised handoffs

After the graph is shown, the deck answers the practical question: how do these agents share information and know what to do? The answer is a single shared state object passed through agent functions with the same signature. As functions update that state, the workflow can route based on fields like category, and it can also decide when tool use should occur.

  • The deck frames this design as specific to LangGraph choices, while still calling it a good pattern.
  • Each agent is modeled as a Python function that consumes the shared state.
  • The example AgentState includes messages, response, and category.
  • Tool use is treated as a workflow decision, not as random side behavior.
