From UI Chat to API Basics

This section covers the first practical transition in the session: moving from a chat interface into explicit model calls, credentials, token awareness, and simple notebook setup.

Study

The point was not "learn Python first"

The instructor used a deliberately small API example to show what a chat UI usually hides. In code, you choose a model, pass the input, add instructions, provide an API key, and inspect the response object directly. The session also briefly revisited tokens so learners could remember that the model processes token IDs rather than intact human sentences.

  • Code exposes control that a UI often hides: model choice, instructions, output handling, and future automation hooks.
  • An API key is a credential tied to account access and billing, so storing it as a secret is safer than hardcoding it.
  • Tokens are units of vocabulary, not always full words; they can be words, subwords, or individual characters.
  • The Colab walkthrough emphasized ordinary setup discipline: make a copy, save the secret, grant access, and run the setup cells in order.

Prompt Layers and Structured Output

This section focuses on the most important design move in the middle of the session: treating prompts as layered instructions and forcing the model to return useful structure instead of just prose.

Study

The model sees a stack of instructions, not a single pure question

The instructor explained that a model call can contain a system prompt, a user prompt, and sometimes assistant context, with different levels of authority. The class also looked at structured extraction, where the model was asked to return specific typed fields from a news article instead of a free-flowing summary.

  • The system prompt is the highest-priority behavior layer and is the natural place for role-setting and guardrails.
  • The assistant layer can provide useful context, but it is less authoritative than the system or user prompt.
  • Structured output matters because downstream software needs predictable fields, not only polished prose.
  • Prompt hierarchy creates control, but it does not make generative systems perfectly rule-bound or fully deterministic.

03

Tools, Fresh Data, and Notebook Workflow

This section covers what tools actually add to an LLM system, why freshness matters, and how the live Colab exercise turned abstract ideas into a usable workflow.

Study

Tools are outside the model, but they change what the model can do

The weather example made the limit of base-model knowledge visible: without fresh external information, the model cannot answer current questions reliably. The instructor explained tools as a way to fetch relevant context before the final prompt reaches the model. The notebook walkthrough then showed the non-glamorous side of real usage: secrets, setup order, permissions, and debugging.

  • Freshness problems often come from model cutoff dates, not from bad wording.
  • Tools such as web search, external APIs, and databases add context that gets folded into the final prompt.
  • Reasoning level is another control surface; in the example, low reasoning meant a faster, less loopy path.
  • Many live failures were ordinary workflow problems: wrong tab, missing notebook copy, secrets not toggled, or setup cells not rerun.
