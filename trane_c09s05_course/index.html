From Chat UI to API Control

The session shifts from "using ChatGPT" as a UI habit to treating an LLM like a programmable service with explicit inputs, credentials, and constraints.

Why the session moved from recap into code

The instructor spent a few minutes reconnecting tokens, model size, and training cost to a practical question: if a model only has broad internet knowledge, how do you adapt it for your own use case?

U

UI Thinking

Type a question, get a reply, and mostly accept the hidden defaults.

A

API Thinking

Explicitly choose the model, pass the user input, and add instructions and tools in code.

P

Practical Goal

Make the model useful for software and workplace tasks instead of leaving it as a generic chat box.

A classroom conversation becomes a system diagram

In the room, learners were already comparing model quality, routing, tokens, and odd failure loops. The API lesson turns those frustrations into concrete levers you can control.

The first API example is intentionally small

The class used a minimal sentiment example to isolate the moving parts. That is a teaching trick: start with a tiny job so the mechanics stay visible.

Code Shape 
client = OpenAI(api_key=API_KEY)
response = client.responses.create(
    model="gpt-5-mini",
    instructions="Classify sentiment.",
    input="I'm feeling awful.",
)
Plain English

Create a client object that knows which account is paying for the call.

Choose a model explicitly instead of accepting whatever a UI picked for you.

Pass hidden behavior guidance through instructions.

Pass the actual user content separately as input.

Capture the returned object so code can inspect or reuse it later.

API key Required for authentication and billing. If it is wrong, the model will not answer.
Model You can route requests intentionally instead of accepting a product default.
Input The visible request content, similar to what you type into a chat box.
Tokens The model ultimately sees token IDs, not intact words or human intent.
!
Operational boundary

The instructors shared a public training key for the session, but they were explicit: do not send proprietary Trane information through that external API.

02

Prompt Layers and Structured Output

The class moves past a simple API call and introduces two ideas that make LLMs operationally useful: prompt hierarchy and predictable response structure.

The model is not seeing "just your question"

The instructor broke prompting into layers so learners could see why behavior changes even when the user message stays the same.

1

System Prompt

Highest-priority instructions. This is where tone, role, and guardrails live.

2

User Prompt

The visible request content. It asks for a task but does not outrank the system prompt.

3

Assistant Context

Helpful context can be inserted here, but it carries less authority than the first two layers.

Structured output is what makes downstream software possible

The session's strongest practical point is simple: free-flowing paragraphs are hard to pipe into other systems, but machine-readable fields are easy.

Code Shape 
class NewsExtraction(BaseModel):
    lottery: str
    jackpot: int
    organizer: str

response = client.chat.completions.parse(
    response_format=NewsExtraction,
    ...
)
Plain English

Define the shape of the answer before the model writes it.

Tell the model that the response must contain named fields with specific data types.

Give the model content to analyze, such as a news article.

Receive something your code can reliably inspect without regex guesswork.

Turn the model from a text generator into a software component that feeds later steps.

Hierarchy is both power and risk

The same layering that makes LLMs controllable also creates failure modes. Conflicting instructions, prompt injection, and imperfect guardrails all live here.

i
Key takeaway

If the system prompt says "only output positive or negative," the model should classify sentiment even when the user asks a different question. That is hierarchy in action.

System > User > Assistant That was the practical priority order the instructor emphasized.
Default behavior exists Even if you omit instructions, the API still has some unseen defaults.
Guardrails are hard Generative systems can receive mixed signals, and hard constraints are still an active design problem.

03

Tools, Fresh Data, and Notebook Practice

The closing arc of the session shows why tools matter, how they compensate for model limits, and what a workable classroom setup looks like in Colab.

The weather question exposes the model's limits

When asked for today's forecast, the model fails without a tool. That is not a bug in wording. It is a limitation of stale training data and cutoff dates.

C

Cutoff Date

The base model cannot know what happened after its training knowledge stopped.

T

Tool Access

Web search, weather APIs, and databases let the system pull fresh context.

R

Reasoning Level

The demo kept reasoning on low so the model would answer directly instead of looping too much.

A tool is really prompt composition plus extra context

One of the clearest explanations in the session is that the model itself does not "know web search." The orchestration layer fetches outside information, then packages it into the prompt the model sees.

Conceptual Flow 
question -> web search tool
tool result -> added context
instructions + user input + tool result
-> one composed prompt
-> model answer
Plain English

The system first notices that the question needs information the base model does not have.

A tool call fetches the missing data from the web, an API, or a database.

That fresh data is blended with the prompt instructions.

The model ultimately receives one larger context packet, not a magical new brain stem.

That is why the instructor called tools an "add-on brain" only in a temporary, situational sense.

The notebook workflow is procedural on purpose

The hands-on part of the session was less about Python mastery and more about making the setup repeatable for people who do not live in code every day.

Colab Checklist The minimum sequence needed for the demo to work
1. Make your own copy Otherwise edits vanish when the notebook closes.
2. Open Secrets Store the API key as a named secret instead of pasting it into code.
3. Toggle notebook access Grant the runtime permission to read that secret.
4. Run setup cells first Skipping imports or environment setup leads to errors like missing names.
?
What the debugging examples were teaching

Many notebook failures in the transcript were not "AI problems." They were setup-order problems: wrong tab, no copy, secret not toggled, or top cells not rerun.
