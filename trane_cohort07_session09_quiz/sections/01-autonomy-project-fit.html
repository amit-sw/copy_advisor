Autonomy, Guardrails, and Project Fit

This section focuses on when to let agents act on their own, and how that choice shapes project design.

Study

The session's first new idea was not "more autonomy at all costs"

After a brief recap, the instructor shifted into a fresh discussion about what changes when you move from a workflow agent to a deep agent. In a workflow, the developer decides the graph and the sequence of steps. In a deep agent, the developer still chooses the tools and sets the high-level goal, but the agent gets more freedom to plan the sequence and decide which tools to call.

  • Autonomy can be useful, but learners repeatedly emphasized the need for guardrails, especially for higher-impact tasks.
  • Different components may deserve different autonomy levels; the right design depends on domain, risk, and expected outcomes.
  • Project ideas were judged partly by feasibility: API access, safe use of internal data, and whether public or synthetic data could stand in for classroom discussion.
  • The session encouraged practical project scoping: start with a workable pattern, avoid sharing Trane IP, and use one-on-one sessions to refine implementation details.
