01

Autonomy, Guardrails, and Project Fit

This section focuses on when to let agents act on their own, and how that choice shapes project design.

Study

The session's first new idea was not "more autonomy at all costs"

After a brief recap, the instructor shifted into a fresh discussion about what changes when you move from a workflow agent to a deep agent. In a workflow, the developer decides the graph and the sequence of steps. In a deep agent, the developer still chooses the tools and sets the high-level goal, but the agent gets more freedom to plan the sequence and decide which tools to call.

  • Autonomy can be useful, but learners repeatedly emphasized the need for guardrails, especially for higher-impact tasks.
  • Different components may deserve different autonomy levels; the right design depends on domain, risk, and expected outcomes.
  • Project ideas were judged partly by feasibility: API access, safe use of internal data, and whether public or synthetic data could stand in for classroom discussion.
  • The session encouraged practical project scoping: start with a workable pattern, avoid sharing Trane IP, and use one-on-one sessions to refine implementation details.

02

From Fixed Workflow to Tool Use

This section covers how the stock-analysis workflow was upgraded from pure prompting to mock and real tool calls.

Study

Tool access increased realism, but orchestration still stayed explicit

After the break, the session returned to the stock-analysis example. The baseline workflow had three specialist agents with different prompts and a fourth synthesis agent that combined their structured outputs into a final report. The next notebooks did not turn that system into a deep agent yet. Instead, they progressively added tool usage.

  • The baseline design sent the same user request to fundamental, technical, and risk agents, each constrained to a fixed schema.
  • Notebook 6 introduced mock tool calls using fake databases to teach the pattern without adding complex external dependencies.
  • Notebook 7 replaced the fake tools with real wrappers such as Yahoo Finance and pandas, while keeping the overall sequence under developer control.
  • Tools often require structured inputs like a ticker symbol, which exposed a usability gap between natural-language user requests and machine-facing tool parameters.

03

Normalization, Shared Context, and Deeper Agents

This section follows the next refinement steps: extracting tool inputs, improving payloads, and preparing for deeper autonomy.

Study

The workflow became easier for users and more structured for developers

Once real tools required a ticker symbol, the session moved to request normalization. The idea was to preserve a natural user experience while still giving tools the structured inputs they need. Then the discussion shifted toward more durable shared context and toward the later notebooks where the workflow edges closer to a deep-agent pattern.

  • Notebook 8 added a preprocessing step that tries to resolve the stock ticker or company name from a free-form request before specialists run.
  • This richer request context is a kind of prompt engineering: instead of dumping the raw question into the model, the system prepares a better input.
  • The orchestration was wrapped into one higher-level function so the user makes one call while the system handles normalization, tool calls, and synthesis.
  • Notebook 9 tightened shared memory by moving from a loose Python dictionary to a structured Pydantic class, which better supports multi-agent coordination.
