Notebook UX And Debugging

The last layer turns all of the backend behavior into something a presenter can operate live: widgets for input, structured renderers for outputs, and a trace log that exposes what the system just did.

One click fans out into three possible runs

The notebook UI is built with ipywidgets. The user picks a mode, types a question, clicks Run Demo, and the handler decides which runner functions to execute.

1
Read the current controls

Mode dropdown, query text area, and Top K slider define the run configuration.

2
Reset trace state

The notebook clears old trace events so each run tells one clean story.

3
Execute selected modes

All modes can run, or the presenter can isolate baseline, RAG, or the agent.

4
Render answer plus diagnostics

Warnings, evidence, climate profile, trace, and final answer are displayed in a consistent order.

The click handler is the notebook conductor

The handler is small, but it coordinates nearly everything. It validates the query, chooses the active mode, resets traces between sections, and delegates rendering to the specialized helpers.

🔍
Trace log is not chain of thought

The trace records observable system events like “retrieval invoked” or “climate profile received.” It is operational telemetry, not hidden reasoning text.

What to inspect when the demo behaves strangely

Good demos are debuggable demos. This notebook already gives you strong inspection surfaces if you know the order to use them.

⚠️

Warnings first

Look for placeholder vector store messages, retrieval failures, or climate lookup warnings before blaming the model.

🧾

Evidence second

If the retrieved snippets are weak, the final answer will also be weak even if the model is behaving correctly.

🪵

Trace third

The ordered trace tells you which path ran and which tool calls actually fired.

render_evidence() Check whether the retrieval layer actually found useful snippets.
render_climate_profile() Inspect resolved location, temperature ranges, and maintenance notes when location matters.
render_trace() Verify the execution path without exposing hidden model reasoning.
