What This Demo Is Proving

This notebook is a side-by-side lab for three answer styles: plain model output, retrieval-grounded output, and a tool-using agent that adds climate context when a location appears in the query.

Bootstrapping is defensive, not fancy

The notebook first checks normal environment variables, then falls back to Colab secrets. That keeps the demo portable between notebook sessions and regular Python execution.

Code 
def get_secret(name: str, default: str = "") -> str:
    if os.getenv(name):
        return os.getenv(name)
    if userdata is not None:
        try:
            value = userdata.get(name)
            if value:
                return value
        except Exception:
            pass
    return default
Plain English

Create one helper whose whole job is to fetch a named secret.

If the value already exists in the normal process environment, trust that first.

If we are inside Colab, try its secret store next.

If the secret exists there, return it and keep moving.

If Colab secret access fails, do not crash on that step alone.

Fall back to a default value so the rest of setup can decide whether that is acceptable.

💡
Key idea

This file is not trying to be a production system. It is trying to make architectural differences visible, so every feature is bent toward explainability.

How RAG Grounds the Answer

Retrieval-augmented generation is the notebook’s evidence lane. It turns a free-form HVAC question into a search against the vector store, then feeds the resulting snippets into a stricter prompt.

RAG is a courier route, not magic

Picture a messenger carrying the user question to a document index, then carrying the best snippets back to the model. The answer becomes more trustworthy because the path is explicit.

The retrieval helper is intentionally literal

Notice what the code does not do. It does not blend hidden heuristics into the answer. It builds search arguments, calls vector search, normalizes the response, and records warnings when evidence is missing.

Code 
kwargs = {
    "vector_store_id": OPENAI_VECTOR_STORE_ID,
    "query": query,
    "max_num_results": int(top_k),
    "rewrite_query": True,
}

try:
    response = openai_client.vector_stores.search(**kwargs)
    results = normalize_search_results(response)
    add_trace(f"Documents found: {len(results)} chunk(s) from vector store {OPENAI_VECTOR_STORE_ID}")
except Exception as exc:
    warning = f"Retrieval failed: {exc}"
Plain English

Build one clean request object for the vector store.

Aim the search at the configured HVAC corpus, not some hardcoded local file.

Pass the user question through directly.

Respect the chosen Top K so the presenter can control result breadth.

Allow query rewriting so search can better match the phrasing stored in the corpus.

Run the search, convert the response into plain Python structures, and log how many chunks came back.

If the retrieval call fails, convert that failure into a visible warning instead of a silent mystery.

🧭
Grounded does not mean perfect

RAG can still retrieve weak evidence. The important improvement is that the weakness becomes inspectable instead of hidden inside pure model confidence.

The RAG prompt narrows the contract

Prompting does not just style the answer. Here it changes the rules: answer only from retrieved evidence, admit incompleteness, and cite filenames in square brackets.

1
Retrieve snippets

The helper returns scored chunks plus a text bundle that can be dropped into the prompt.

2
Constrain the model

The system message says the answer must stay inside the retrieved evidence boundary.

3
Render evidence separately

The notebook shows the retrieved rows and the final answer in different sections, which makes debugging much easier.

📄

Turns SDK-shaped data into a predictable list of filenames, scores, attributes, and snippets.

⚠️

Makes empty retrieval or exceptions visible to the presenter before the answer is interpreted.

The agent path is where the demo gets interesting. It can notice a place name, fetch climate history for that location, and then still require document evidence before making product claims.

The agent has two tools and one traffic law

The traffic law is simple: climate is advisory, documents are authoritative. That keeps weather context useful without letting it replace grounded product evidence.

Climate context is derived from history, not guessed

This function does more than geocode a city. It pulls ten years of archive weather, groups by month, converts units, and then turns those aggregates into maintenance notes a human can actually use.

Code 
climate_df = pd.DataFrame({
    "date": pd.to_datetime(daily["time"]),
    "tmax_c": daily["temperature_2m_max"],
    "tmin_c": daily["temperature_2m_min"],
    "precip_mm": daily.get("precipitation_sum", [None] * len(daily["time"])),
})
climate_df["month"] = climate_df["date"].dt.month

monthly = climate_df.groupby("month", as_index=False).agg(
    avg_high_c=("tmax_c", "mean"),
    avg_low_c=("tmin_c", "mean"),
)
Plain English

Take the raw weather arrays and build a tidy DataFrame.

Convert the date strings into real dates so month-based math becomes easy.

Store daily high temperature, low temperature, and rainfall in named columns.

If rainfall is missing, pad the column with empty values so the table shape still works.

Add a month column so each day now knows which monthly bucket it belongs to.

Group all rows by month and compute average highs and lows for each month.

Those monthly summaries later become human-facing climate and maintenance notes.

⚠️
Important boundary

The agent prompt explicitly says climate is advisory only. A hot city can influence the recommendation story, but it cannot prove which product supports BACnet or monitoring features.

Notebook UX And Debugging

The last layer turns all of the backend behavior into something a presenter can operate live: widgets for input, structured renderers for outputs, and a trace log that exposes what the system just did.

One click fans out into three possible runs

The notebook UI is built with ipywidgets. The user picks a mode, types a question, clicks Run Demo, and the handler decides which runner functions to execute.

1
Read the current controls

Mode dropdown, query text area, and Top K slider define the run configuration.

2
Reset trace state

The notebook clears old trace events so each run tells one clean story.

3
Execute selected modes

All modes can run, or the presenter can isolate baseline, RAG, or the agent.

4
Render answer plus diagnostics

Warnings, evidence, climate profile, trace, and final answer are displayed in a consistent order.

The click handler is the notebook conductor

The handler is small, but it coordinates nearly everything. It validates the query, chooses the active mode, resets traces between sections, and delegates rendering to the specialized helpers.

Code 
def on_run_clicked(_):
    with output:
        clear_output(wait=True)
        TRACE_LOG.clear()
        try:
            query = query_widget.value.strip()
            if not query:
                raise ValueError("Enter a query before running the demo.")
            top_k = int(top_k_widget.value)
            display(Markdown("# HVAC Demo Output"))
            display(Markdown(f"**Selected Mode**: `{mode_widget.value}`"))
Plain English

Define the callback that runs when the user presses the notebook button.

Render everything into the shared output panel.

Clear the old UI first so the new run does not stack on top of stale results.

Reset the trace so this run has its own timeline.

Try the full run inside error handling so mistakes become readable notebook output.

Pull the current query text from the text area and trim whitespace.

If the query is empty, stop immediately with a human-readable error.

Read the current Top K slider and show a heading that explains what mode is being demonstrated.

🔍
Trace log is not chain of thought

The trace records observable system events like “retrieval invoked” or “climate profile received.” It is operational telemetry, not hidden reasoning text.

What to inspect when the demo behaves strangely

Good demos are debuggable demos. This notebook already gives you strong inspection surfaces if you know the order to use them.

⚠️

Warnings first

Look for placeholder vector store messages, retrieval failures, or climate lookup warnings before blaming the model.

🧾

Evidence second

If the retrieved snippets are weak, the final answer will also be weak even if the model is behaving correctly.

🪵

Trace third

The ordered trace tells you which path ran and which tool calls actually fired.

render_evidence() Check whether the retrieval layer actually found useful snippets.
render_climate_profile() Inspect resolved location, temperature ranges, and maintenance notes when location matters.
